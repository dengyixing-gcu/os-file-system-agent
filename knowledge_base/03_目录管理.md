# 6.3 目录管理

**目录（Directory）** 是一种特殊的文件，用于记录文件名与文件物理地址的映射关系，实现**按名存取**。

---

## 一、文件控制块（FCB）

### 1. FCB 的定义

**文件控制块（File Control Block, FCB）** 是操作系统为每个文件建立的一个数据结构，用于存储文件的属性和管理信息。**FCB 是文件存在的唯一标志**。

```
┌─────────────────────────────────────────────────┐
│              FCB 数据结构                        │
├─────────────────────────────────────────────────┤
│  基本信息区：                                   │
│  ├─ 文件名（主名 + 扩展名）                     │
│  ├─ 文件物理地址（起始块号/索引表地址）          │
│  ├─ 文件长度（字节数/块数）                     │
│  ├─ 文件类型（普通/目录/设备/链接）             │
│  │                                              │
│  存取控制信息区：                               │
│  ├─ 文件所有者 ID                               │
│  ├─ 访问权限（读/写/执行）                      │
│  ├─ 口令/加密信息                               │
│  │                                              │
│  使用信息区：                                   │
│  ├─ 创建时间                                    │
│  ├─ 最后修改时间                                │
│  ├─ 最后访问时间                                │
│  ├─ 当前打开次数                                │
│  └─ 当前读写位置                                │
└─────────────────────────────────────────────────┘
```

### 2. FCB 的大小

典型 FCB 大小约 **32~64 字节**，包含约 30~50 个属性项。

---

## 二、索引节点（inode）

### 1. 为什么需要 inode？

**问题**：FCB 包含文件名，而目录查询时需要频繁读取 FCB。如果 FCB 很大，会导致：
- 磁盘 I/O 次数增加
- 内存占用大
- 目录查询效率低

**解决方案**：将 FCB 拆分为两部分
- **索引节点（inode）**：存储文件属性（除文件名外）
- **目录项**：只存储文件名和 inode 号

```
┌─────────────────────────────────────────────────┐
│          FCB 拆分前后对比                        │
│                                                 │
│  拆分前：                                        │
│  目录项 = [文件名 | 完整 FCB(64 字节)]            │
│                                                 │
│  拆分后：                                        │
│  目录项 = [文件名 | inode 号 (4 字节)]           │
│  inode  = [文件属性 (60 字节)]                   │
│                                                 │
│  效果：目录项从 64 字节 → 约 20 字节，缩小 3 倍     │
└─────────────────────────────────────────────────┘
```

### 2. inode 结构

```
┌─────────────────────────────────────────────────┐
│              inode 数据结构                      │
├─────────────────────────────────────────────────┤
│  文件类型和访问权限                             │
│  文件所有者 UID 和组 ID GID                     │
│  文件大小                                       │
│  时间戳（创建/修改/访问）                       │
│  链接计数（硬链接数）                           │
│  指针：                                         │
│    直接地址 [0~9]  ──→ 数据块                   │
│    一次间接 [10]   ──→ 索引块                   │
│    二次间接 [11]   ──→ 一级索引 → 二级索引      │
│    三次间接 [12]   ──→ 多级索引链               │
└─────────────────────────────────────────────────┘
```

### 3. inode 与 FCB 对比

| 对比项 | FCB | inode |
|--------|-----|-------|
| **包含文件名** | 是 | 否 |
| **存储位置** | 目录文件中 | 独立 inode 区 |
| **查找方式** | 遍历目录 | inode 号直接定位 |
| **典型系统** | MS-DOS | Unix/Linux |
| **目录项大小** | 大 | 小 |

## 三、目录结构

### 1. 单级目录（Single-Level Directory）

**结构**：整个文件系统只有一张目录表，所有文件都在同一目录下。

```
┌─────────────────────────────────────────────────┐
│            单级目录示意图                        │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │           主目录（Master Directory）     │    │
│  ├─────────────────────────────────────────┤    │
│  │  文件名  │  FCB/地址  │  属性            │    │
│  ├─────────────────────────────────────────┤    │
│  │  file1   │   0x100   │  rwx             │    │
│  │  file2   │   0x200   │  rw-             │    │
│  │  file3   │   0x300   │  r--             │    │
│  │  ...     │   ...     │  ...             │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  特点：所有用户共享同一目录                      │
└─────────────────────────────────────────────────┘
```

**优点**：
- ✅ 结构简单
- ✅ 实现容易
- ✅ 目录查询快（文件少时）

**缺点**：
- ❌ **不允许文件重名**（最大问题）
- ❌ 无法实现文件共享
- ❌ 不适合多用户环境
- ❌ 文件数量多时查询效率低

**适用场景**：早期单用户操作系统（如 CP/M）

---

### 2. 两级目录（Two-Level Directory）

**结构**：分为**主文件目录（MFD）**和**用户文件目录（UFD）**。

```
┌─────────────────────────────────────────────────┐
│            两级目录示意图                        │
│                                                 │
│              主文件目录 (MFD)                   │
│          ┌─────────────────────────┐            │
│          │ 用户名 │ UFD 地址 │     │            │
│          ├─────────────────────────┤            │
│          │ user1  │  0x100  │     │            │
│          │ user2  │  0x200  │     │            │
│          │ user3  │  0x300  │     │            │
│          └────┬────┴────┬────┴────┘            │
│               │          │                      │
│               ↓          ↓                      │
│        ┌──────────┐ ┌──────────┐               │
│        │ UFD1     │ │ UFD2     │               │
│        ├──────────┤ ├──────────┤               │
│        │file1     │ │file1     │               │
│        │file2     │ │report.txt│               │
│        │data.txt  │ │prog.c    │               │
│        └──────────┘ └──────────┘               │
│                                                 │
│  特点：每个用户有独立目录，不同用户可重名         │
└─────────────────────────────────────────────────┘
```

**优点**：
- ✅ **允许不同用户文件重名**
- ✅ 支持多用户环境
- ✅ 一定程度的隔离保护

**缺点**：
- ❌ 同一用户下仍不能重名
- ❌ 不便于文件共享
- ❌ 无法反映文件间的逻辑关系

---

### 3. 多级目录（树形目录）

**结构**：目录本身也是文件，可以包含子目录，形成树形结构。

```
┌─────────────────────────────────────────────────┐
│          树形目录结构示意图                      │
│                                                 │
│                    / (根目录)                   │
│                   /│\                           │
│                  / │ \                          │
│                /   │   \                        │
│              /     │     \                      │
│            bin    home   etc                    │
│            /│    /   \    │                     │
│           / │   /     \   │                     │
│         ls cp alice  bob  passwd                │
│               / | \    │                       │
│              /  |  \   │                       │
│           docs src mail report.txt              │
│           /    │                                 │
│         os   main.c                              │
│                                                 │
│  路径示例：                                      │
│  /home/alice/docs/os/notes.txt                  │
└─────────────────────────────────────────────────┘
```

**优点**：
- ✅ **同一用户下也可重名**（只要路径不同）
- ✅ 便于文件分类管理
- ✅ 支持灵活的共享机制
- ✅ 符合用户思维习惯

**缺点**：
- ❌ 结构复杂
- ❌ 路径名解析开销大

**典型应用**：Unix/Linux、Windows、macOS 等现代操作系统

## 四、路径名

### 1. 绝对路径（Absolute Path）

**定义**：从**根目录**开始的完整路径。

```
┌─────────────────────────────────────────────────┐
│              路径名示例                          │
│                                                 │
│  目录树：                                        │
│       /                                         │
│      / \                                        │
│    home  bin                                    │
│    / \                                          │
│  alice bob                                      │
│   |                                             │
│  docs                                           │
│   |                                             │
│  report.txt                                     │
│                                                 │
│  绝对路径：/home/alice/docs/report.txt          │
│  （始终以 / 开头）                               │
└─────────────────────────────────────────────────┘
```

**特点**：
- 始终以根目录 `/`（Unix）或盘符 `C:\`（Windows）开头
- 唯一确定一个文件
- 任何位置都可用

---

### 2. 相对路径（Relative Path）

**定义**：从**当前目录**开始的路径。

```
┌─────────────────────────────────────────────────┐
│           相对路径示例                           │
│                                                 │
│  假设当前目录：/home/alice                      │
│                                                 │
│       /                                         │
│      / \                                        │
│    home  bin                                    │
│    / \                                          │
│  alice bob    ← 当前目录 (.)                    │
│   |                                             │
│  docs                                           │
│   | \                                           │
│   |  os                                         │
│  report.txt                                     │
│                                                 │
│  相对路径表示：                                  │
│  docs/report.txt    → 当前目录下的 docs/report.txt│
│  ./docs/report.txt  → 同上（.表示当前目录）      │
│  ../bob/file.txt    → 上一级的 bob/file.txt     │
│  /bin/ls            → 绝对路径（以/开头）       │
└─────────────────────────────────────────────────┘
```

**特殊符号**：
| 符号 | 含义 |
|------|------|
| `.` | 当前目录 |
| `..` | 父目录（上一级） |
| `~` | 用户主目录 |

---

### 🔄 绝对路径 vs 相对路径 对比

| 对比项 | 绝对路径 | 相对路径 |
|--------|----------|----------|
| **起点** | 根目录 | 当前目录 |
| **开头** | `/` 或盘符 | 文件名或 `.`/`..` |
| **唯一性** | 唯一 | 依赖当前目录 |
| **长度** | 较长 | 较短 |
| **可移植性** | 差（依赖目录结构） | 好 |
| **示例** | `/home/alice/file.txt` | `./file.txt` |

## 五、目录查询技术

### 1. 线性检索法（顺序查找）

**方法**：从头到尾遍历目录项，逐个比较文件名。

```
┌─────────────────────────────────────────────────┐
│          线性检索示意图                          │
│                                                 │
│  目录文件：                                      │
│  ┌──────┬──────┬──────┬──────┬──────┐          │
│  │ item0│ item1│ item2│ item3│ item4│          │
│  │ file1│ file2│ file3│ file4│ file5│          │
│  └──────┴──────┴──────┴──────┴──────┘          │
│     ↑                                           │
│     查找 file3:                                 │
│     1. 比较 item0 → 不匹配                      │
│     2. 比较 item1 → 不匹配                      │
│     3. 比较 item2 → 匹配！返回 inode 号          │
│                                                 │
│  平均查找长度：(n+1)/2                          │
└─────────────────────────────────────────────────┘
```

**优点**：
- ✅ 实现简单
- ✅ 适合小目录

**缺点**：
- ❌ 大目录效率低（O(n)）

---

### 2. 哈希检索法

**方法**：使用哈希函数将文件名映射到哈希表位置。

```
┌─────────────────────────────────────────────────┐
│          哈希检索示意图                          │
│                                                 │
│  哈希函数：H(filename) = filename 的 ASCII 和 % 表长│
│                                                 │
│  H("file1") = (102+105+108+101+49) % 7 = 5      │
│  H("file2") = (102+105+108+101+50) % 7 = 6      │
│  H("file3") = (102+105+108+101+51) % 7 = 0      │
│                                                 │
│  哈希表：                                        │
│  ┌───┬───┬───┬───┬───┬───┬───┐                 │
│  │ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │                 │
│  ├───┼───┼───┼───┼───┼───┼───┤                 │
│  │file3│   │   │   │   │file1│file2│           │
│  └───┴───┴───┴───┴───┴───┴───┘                 │
│                                                 │
│  查找 file1: H("file1")=5 → 直接访问位置 5      │
│  时间复杂度：O(1)                               │
└─────────────────────────────────────────────────┘
```

**优点**：
- ✅ 查找速度快（O(1)）
- ✅ 适合大目录

**缺点**：
- ❌ 需要处理哈希冲突
- ❌ 实现复杂

## 六、目录操作

### 常见目录操作：

| 操作 | 系统调用 | 说明 |
|------|----------|------|
| **创建目录** | `mkdir()` | 创建新目录 |
| **删除目录** | `rmdir()` | 删除空目录 |
| **打开目录** | `opendir()` | 打开目录用于读取 |
| **读取目录** | `readdir()` | 读取目录项 |
| **关闭目录** | `closedir()` | 关闭目录 |
| **改变目录** | `chdir()` | 切换当前工作目录 |
| **获取路径** | `getcwd()` | 获取当前工作目录路径 |

### 操作示例（Unix）：

```bash
# 创建目录
mkdir /home/alice/docs

# 进入目录
cd /home/alice/docs

# 查看当前目录
pwd

# 列出目录内容
ls -la

# 删除目录（必须为空）
rmdir /home/alice/docs
```

## 七、易混淆点对比

### 🔄 FCB vs inode

| 对比项 | FCB | inode |
|--------|-----|-------|
| **文件名存储** | 在 FCB 内 | 不在 inode 内 |
| **目录项内容** | 完整 FCB | 文件名 + inode 号 |
| **查找方式** | 顺序遍历目录 | inode 号直接索引 |
| **典型系统** | FAT、早期系统 | Unix、Linux、ext |

### 🔄 硬链接 vs 软链接

| 对比项 | 硬链接 | 软链接（符号链） |
|--------|--------|-----------------|
| **实现方式** | 共享 inode | 存储目标路径 |
| **跨文件系统** | 不支持 | 支持 |
| **链接目录** | 不支持 | 支持 |
| **原文件删除** | 仍可访问 | 链接失效 |
| **本质** | 同一文件多个名字 | 快捷方式 |

## 八、典型例题

**例题 1**：为什么 Unix 系统要将 FCB 拆分为 inode 和目录项？

**答案**：
1. 目录查询时需要频繁读取目录文件
2. FCB 包含文件名和大量属性，体积较大（约 64 字节）
3. 拆分后目录项只包含文件名和 inode 号（约 20 字节）
4. 同样大小的磁盘块可容纳更多目录项
5. 减少磁盘 I/O 次数，提高查询效率

**例题 2**：在树形目录结构中，文件 `/home/alice/docs/report.txt` 和 `/home/bob/docs/report.txt` 是否为同一文件？

**答案**：不是同一文件。虽然文件名相同（都是 report.txt），但路径不同，是两个独立的文件。这正是树形目录的优点——允许文件重名。

---

> ⚠️ **注意**：不同操作系统的目录实现可能有所不同，但基本原理相同。
