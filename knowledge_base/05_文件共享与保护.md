# 6.5 文件共享与保护

---

## 一、文件共享

**文件共享**是指多个用户或进程可以共同访问同一个文件。

### 1. 共享的好处

- ✅ 节省存储空间（多个用户共用一份数据）
- ✅ 便于协作（多人编辑同一文档）
- ✅ 数据一致性（避免多份副本不一致）
- ✅ 简化维护（只需维护一份文件）

---

### 2. 基于索引节点的共享（硬链接）

**原理**：多个目录项指向**同一个 inode**。

```
┌─────────────────────────────────────────────────┐
│        基于索引节点的共享示意图                  │
│                                                 │
│  用户 alice 的目录：                            │
│  ┌─────────────────┐                            │
│  │ report.txt │ 100│──┐                         │
│  └─────────────────┘  │                         │
│                       ↓                         │
│                    ┌─────┐                      │
│  用户 bob 的目录：   │inode│                      │
│  ┌─────────────────┐│ 100 │                      │
│  │ share.txt  │ 100│──┤     │                      │
│  └─────────────────┘│     │                      │
│                     └─────┘                      │
│                       │                          │
│                       ↓                          │
│                   数据块                          │
│                                                 │
│  特点：                                          │
│  • alice/report.txt 和 bob/share.txt 是同一文件  │
│  • inode 中的链接计数 = 2                        │
│  • 任一用户修改，另一用户可见                    │
└─────────────────────────────────────────────────┘
```

**实现机制**：
1. inode 中设置**链接计数（link count）**
2. 每次创建硬链接，计数 +1
3. 删除文件时，计数 -1
4. 计数为 0 时，才真正删除文件

**优点**：
- ✅ 访问效率高（直接通过 inode 访问）
- ✅ 节省存储空间
- ✅ 数据一致性好

**缺点**：
- ❌ 不能跨文件系统（inode 号只在同一文件系统内有效）
- ❌ 不能链接目录（可能导致循环）
- ❌ 原文件删除后，其他链接仍可访问（可能误以为文件已删除）

**命令示例（Unix）**：
```bash
# 创建硬链接
ln /home/alice/report.txt /home/bob/share.txt

# 查看链接数
ls -l
# -rw-r--r-- 2 alice alice 4096 Mar 26 10:00 report.txt
#            ↑
#        链接数=2
```

---

### 3. 基于符号链的共享（软链接）

**原理**：创建一个**链接文件**，存储目标文件的路径名。

```
┌─────────────────────────────────────────────────┐
│         基于符号链的共享示意图                   │
│                                                 │
│  用户 alice 的目录：                            │
│  ┌─────────────────┐                            │
│  │ report.txt │ 100│                            │
│  └─────────────────┘                            │
│        ↑                                        │
│        │ (inode 100)                            │
│        │                                        │
│        ↓                                        │
│    数据块                                      │
│                                                 │
│  用户 bob 的目录：                               │
│  ┌─────────────────┐                            │
│  │ link.txt   │ 200│                            │
│  └─────────────────┘                            │
│        │                                        │
│        │ (inode 200)                            │
│        │                                        │
│        ↓                                        │
│  ┌─────────────────┐                            │
│  │ /home/alice/    │  ← 存储目标路径            │
│  │ report.txt      │                            │
│  └─────────────────┘                            │
│                                                 │
│  特点：                                          │
│  • link.txt 是符号链接文件                       │
│  • 访问时需解析路径                              │
│  • 原文件删除后，链接失效（悬空链接）            │
└─────────────────────────────────────────────────┘
```

**优点**：
- ✅ 可跨文件系统
- ✅ 可链接目录
- ✅ 灵活性高

**缺点**：
- ❌ 访问效率低（需路径解析）
- ❌ 原文件删除后链接失效
- ❌ 占用额外存储空间（存储路径名）

**命令示例（Unix）**：
```bash
# 创建符号链接
ln -s /home/alice/report.txt /home/bob/link.txt

# 查看符号链接
ls -l
# lrwxrwxrwx 1 bob bob 24 Mar 26 10:00 link.txt -> /home/alice/report.txt
# ↑
# 'l' 表示符号链接
```

---

### 🔄 硬链接 vs 软链接 对比

| 对比项 | 硬链接 | 软链接（符号链） |
|--------|--------|-----------------|
| **实现方式** | 共享 inode | 存储路径名 |
| **inode 号** | 相同 | 不同 |
| **跨文件系统** | ❌ 不支持 | ✅ 支持 |
| **链接目录** | ❌ 不支持 | ✅ 支持 |
| **原文件删除** | 仍可访问 | 链接失效 |
| **文件大小** | 与原文件相同 | 路径名长度 |
| **访问效率** | 高 | 低（需解析） |
| **类比** | 同一文件的别名 | Windows 快捷方式 |

## 二、文件保护

文件保护的目标：**防止未经授权的文件访问**。

### 1. 访问控制

#### （1）访问控制矩阵

```
┌─────────────────────────────────────────────────┐
│           访问控制矩阵示意图                      │
│                                                 │
│           │ 文件 A │ 文件 B │ 文件 C │ 文件 D  │
│  ─────────┼───────┼───────┼───────┼────────   │
│  用户 alice│ rwx   │ r--   │ rw-   │ ---      │
│  用户 bob  │ r--   │ rwx   │ r--   │ r--      │
│  用户 carol│ rw-   │ r--   │ ---   │ rwx      │
│  用户 group│ r-x   │ r-x   │ r--   │ r--      │
│                                                 │
│  图例：r=读，w=写，x=执行，-=无权限              │
│                                                 │
│  缺点：矩阵稀疏，浪费空间                        │
└─────────────────────────────────────────────────┘
```

#### （2）访问控制列表（ACL）

为每个文件建立一个访问控制列表。

```
┌─────────────────────────────────────────────────┐
│          访问控制列表（ACL）示例                 │
│                                                 │
│  文件：report.txt                               │
│  ┌─────────────────────────────────────────┐    │
│  │ ACL:                                    │    │
│  │   user:alice : rwx                      │    │
│  │   user:bob   : r--                      │    │
│  │   user:carol : rw-                      │    │
│  │   group:dev  : r-x                      │    │
│  │   group:hr   : r--                      │    │
│  │   mask       : rw-                      │    │
│  │   other      : r--                      │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  优点：灵活，可精确控制每个用户的权限           │
│  缺点：ACL 过长时效率低                          │
└─────────────────────────────────────────────────┘
```

#### （3）Unix 权限模型（rwx）

```
┌─────────────────────────────────────────────────┐
│          Unix 权限模式示意图                     │
│                                                 │
│  -rwxr-xr--  alice  developers  report.txt     │
│  │││││││││     │       │         │            │
│  │││││││││     │       │         └─ 文件名     │
│  │││││││││     │       └─── 所属组             │
│  │││││││││     └─── 所有者                     │
│  │││││││││                                     │
│  │││││││└─ 其他用户权限 (r-- = 只读)           │
│  ││││││└── 组权限 (r-x = 读 + 执行)            │
│  │││││└─── 所有者权限 (rwx = 读 + 写 + 执行)   │
│  │││││                                        │
│  ││││└──── 特殊权限位 (setuid/setgid/sticky)  │
│  ││││                                         │
│  │││└───── 文件类型                           │
│  │││      - = 普通文件                        │
│  │││      d = 目录                            │
│  │││      l = 符号链接                        │
│  │││      c = 字符设备                        │
│  │││      b = 块设备                          │
│  └┴┴────── 共 10 个字符                         │
│                                                 │
│  权限值：                                       │
│  r (读) = 4    w (写) = 2    x (执行) = 1      │
│                                                 │
│  数字表示：                                     │
│  rwx = 4+2+1 = 7    rw- = 4+2 = 6              │
│  r-x = 4+1 = 5    r-- = 4                      │
│  --- = 0                                        │
│                                                 │
│  chmod 755 file.txt  →  rwxr-xr-x              │
│  chmod 644 file.txt  →  rw-r--r--              │
└─────────────────────────────────────────────────┘
```

### 2. 口令保护

**原理**：为文件设置访问口令，用户需输入正确口令才能访问。

```
┌─────────────────────────────────────────────────┐
│           口令保护机制示意图                     │
│                                                 │
│  文件 FCB：                                      │
│  ┌─────────────────────────────────────────┐    │
│  │ 文件名：secret.txt                      │    │
│  │ 口令：encrypted_password_hash           │    │
│  │ 权限：需要口令才能访问                   │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  访问流程：                                      │
│  1. 用户请求访问文件                            │
│  2. 系统提示输入口令                            │
│  3. 系统加密用户输入的口令                      │
│  4. 与 FCB 中存储的哈希值比较                    │
│  5. 匹配则允许访问，否则拒绝                    │
│                                                 │
│  缺点：                                          │
│  • 口令管理困难（每个文件一个口令）             │
│  • 口令可能被破解                               │
│  • 无法区分不同用户的权限                       │
└─────────────────────────────────────────────────┘
```

### 3. 加密保护

**原理**：文件内容加密存储，只有授权用户才能解密。

```
┌─────────────────────────────────────────────────┐
│           文件加密保护示意图                     │
│                                                 │
│  明文文件：                                      │
│  ┌─────────────────────────────────────────┐    │
│  │ This is a secret document.              │    │
│  │ Confidential information inside.        │    │
│  └─────────────────────────────────────────┘    │
│                    ↓ 加密 (AES-256)             │
│  密文文件（磁盘上）：                            │
│  ┌─────────────────────────────────────────┐    │
│  │ X7f#kL9@mP2$vN5&qR8!tY1^wZ4*            │    │
│  │ hJ6%bC3!dF0(gH7)jK2+lM5-nP8=            │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  加密方式：                                      │
│  • 对称加密：加密解密用同一密钥（如 AES）       │
│  • 非对称加密：公钥加密，私钥解密（如 RSA）     │
│                                                 │
│  优点：                                          │
│  • 即使文件被盗也无法读取内容                   │
│  • 安全性高                                     │
│                                                 │
│  缺点：                                          │
│  • 加解密开销大                                 │
│  • 密钥管理复杂                                 │
└─────────────────────────────────────────────────┘
```

## 三、文件权限管理

### 1. 权限分类

| 权限类型 | 对文件的含义 | 对目录的含义 |
|----------|-------------|-------------|
| **读 (r)** | 读取文件内容 | 列出目录内容 |
| **写 (w)** | 修改文件内容 | 在目录中创建/删除文件 |
| **执行 (x)** | 作为程序运行 | 进入目录（cd） |

### 2. 权限设置命令（Unix）

```bash
# 数字方式设置权限
chmod 755 script.sh    # rwxr-xr-x
chmod 644 file.txt     # rw-r--r--
chmod 700 private.key  # rwx------

# 符号方式设置权限
chmod u+x script.sh    # 给所有者添加执行权限
chmod g-w file.txt     # 移除组写权限
chmod o=r file.txt     # 设置其他用户只读
chmod a+x script.sh    # 给所有人添加执行权限

# 递归设置目录权限
chmod -R 755 /var/www

# 查看文件权限
ls -l file.txt
```

### 3. 特殊权限

| 权限 | 符号 | 作用 |
|------|------|------|
| **setuid** | s (所有者) | 执行时以文件所有者身份运行 |
| **setgid** | s (组) | 执行时以文件所属组身份运行 |
| **sticky** | t | 目录中只有文件所有者可删除文件 |

```bash
# setuid 示例：/usr/bin/passwd
-rwsr-xr-x 1 root root /usr/bin/passwd
# ↑ 's' 表示 setuid，普通用户执行时以 root 身份运行

# sticky 示例：/tmp 目录
drwxrwxrwt 10 root root /tmp
# ↑ 't' 表示 sticky，用户只能删除自己的文件
```

## 四、典型例题

**例题 1**：硬链接和软链接的主要区别是什么？

**答案**：
1. **实现方式**：硬链接共享 inode，软链接存储路径名
2. **跨文件系统**：硬链接不支持，软链接支持
3. **链接目录**：硬链接不支持，软链接支持
4. **原文件删除**：硬链接仍可访问，软链接失效
5. **访问效率**：硬链接高，软链接低

**例题 2**：某文件权限为 `-rwxr-xr--`，用数字如何表示？

**答案**：754
- 所有者：rwx = 4+2+1 = 7
- 组：r-x = 4+0+1 = 5
- 其他：r-- = 4+0+0 = 4

**例题 3**：为什么 Unix 系统不允许对目录创建硬链接？

**答案**：
1. 目录硬链接可能导致目录树出现环
2. 破坏树形目录结构
3. 可能导致遍历目录时进入死循环
4. 增加文件系统复杂性
5. 软链接已能满足目录链接需求

> ⚠️ **注意**：文件共享和保护机制因操作系统而异，实际使用时请参考具体系统文档。
